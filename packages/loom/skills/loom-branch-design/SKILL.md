---
name: loom-branch-design
description: Create or develop alternative design variant branches for Loom-backed proposals or design nodes. Use when assigned to explore multiple design directions.
---

Use this skill only when your assignment includes Loom context/reference or explicitly asks for Loom design branching. Skills are execution procedures for assigned agents; do not invoke Loom slash commands from child-agent work unless a human/parent explicitly asks you to reroute.

If working in a multi-Loom project, use the provided `LOOM_DEFAULT`, explicit `-L`, or `loom current` / `loom list` to confirm the target Loom before writing.

1. Fetch context for the proposal/design node.
2. Identify materially different approaches worth preserving.
3. Create or propose variant nodes with clear assumptions, boundaries, and tradeoffs.
4. If useful, ask a parent/lead to dispatch `design-variant-planner` agents for parallel variant development.
5. Record what evidence or criteria would make each variant win or lose.
6. If multiple writers are developing variants, stay within your assigned branch/node/artifact scope and coordinate before editing shared notes.
7. Do not choose a winner implicitly; use a decision node/workflow when ready.
8. Return a mutation summary listing branches/nodes/notes created or updated, files touched if any, blockers, and evidence needed.

Preserve rejected or deferred alternatives with rationale. Prefer simple coherent designs over speculative abstractions or compatibility bridges. Do not edit Loom internals directly.
