# N-0011 Durable Attention and Delivery State

Date: 2026-04-26
Status: implemented

## Summary

Implement file-backed durable attention/delivery records for child completion, block, and error notifications. Build on N-0002 lineage resolver to scope attention by recipient context (runId/runDir/rootSessionId). Prevent re-delivery of handled completion items to the same recipient or root session. Integrate minimally into CLI `wait`, `result`, `watch`, and Pi extension delivery path.

## Changes

### New module: `packages/tango/src/attention.ts`

- Reusable module with file-backed JSONL state under `$TANGO_HOME/attention.jsonl`
- Key: `(canonicalRecipientKey, targetRunDir, eventId)`
- States: `new`, `delivered`, `seen`, `handled`, `dismissed`, `superseded`
- Append-only JSONL with read-time deduplication by key
- Recipient match logic uses **canonical/exact key semantics**:
  - Prefer `runId`, else normalized `runDir`, else `rootSessionId`, else `anonymous`
  - Two agents with the same `rootSessionId` but different `runId`/`runDir` are **isolated**
- Helpers:
  - `upsertAttentionFromEvent(event, recipient)` — create or update from status event; on new record, supersedes older unresolved records for same recipient+target
  - `markDelivered`, `markSeen`, `markHandled`, `markDismissed`, `markSuperseded`
  - `markLatestDoneHandled(recipient, targetRunDir)` — find latest done record and handle it; if no attention record exists yet, scans the event log for the latest done event, upserts it, then marks handled
  - `shouldDeliverEvent(event, recipient)` — suppress handled done, dismissed, superseded
  - `isHandledForRecipient`, `listUnresolvedAttention`
  - `getRecipientContext()` — reads `TANGO_RUN_ID`, `TANGO_RUN_DIR`, `TANGO_ROOT_SESSION_ID`

### CLI integration: `packages/tango/src/cli.ts`

- `tango watch` — upserts attention record and marks `seen` for each matching status event
- `tango wait` — after observing terminal done, calls `markLatestDoneHandled` for each done target
- `tango result` — after reading result, calls `markLatestDoneHandled` for the target

### Pi extension integration: `packages/tango/extensions/pi/index.ts`

- Imports durable attention store from `../../dist/attention.js` instead of duplicating logic
- On event arrival: `upsertAttentionFromEvent`, then `shouldDeliverEvent` gate
- If deliverable: mark `delivered`, queue for Pi message
- On flush: mark `seen` for all queued events
- `done` events already `handled` are suppressed for the same recipient context
- `blocked`/`error` remain visible/deliverable until explicitly dismissed or superseded
- Delivery filtering uses **documented direct-child semantics** (stricter than `eventMatchesLineage`):
  - Requires `parentRunId` match or normalized `parentRunDir` match
  - Only notifies about immediate children, not all lineage matches
  - Path normalization avoids contradicting `eventMatchesLineage` behavior

### Tests: `packages/tango/src/attention.test.ts`

- 20 test cases covering:
  - upsert and update attention records
  - state transitions (delivered, seen, handled, dismissed, superseded)
  - `markLatestDoneHandled` finds correct record
  - `markLatestDoneHandled` creates handled record from event log when no attention record exists yet
  - `done` handled prevents re-delivery
  - `blocked`/`error` remains deliverable after seen
  - per-recipient isolation by runId and rootSessionId
  - same rootSessionId with different runId/runDir does **not** match
  - older unresolved records are superseded by newer events for same recipient+target
  - unresolved attention listing
  - recipient context from environment

### Co-fixes for worktree build

- Exported `listRootSessions` from `server.ts` (used by dashboard-api)
- Fixed duplicate `schemaVersion` spreads in `server.ts` response payloads
- Fixed duplicate property spreads in `rootSessions.test.ts` `baseMeta`

## Decisions

- Append-only JSONL with read-time dedupe rather than rewrite-on-update for simplicity and crash safety
- `shouldDeliverEvent` suppresses `done` only when `handled`; `blocked`/`error` remain visible until dismissed
- Per-recipient isolation uses **exact canonical key** matching to prevent accidental cross-agent visibility
- `markLatestDoneHandled` guarantees a handled tombstone even when no prior attention record was written (scans event log)
- Pi extension imports compiled `dist/attention.js` at runtime rather than inlining logic; this is safe because the extension already depends on `dist/cli.js`
- Pi delivery intentionally uses direct-child filtering rather than full lineage matching to keep notifications scoped to immediate children

## Re-review fix (2026-04-26)

**Blocker:** Pi batching could deliver superseded/stale attention events and marked `seen` on events that were never actually sent.

**Fix in `packages/tango/extensions/pi/index.ts`:**
- Before `flushTangoEvents` sends `pendingEvents`, it now re-filters each event against the current `shouldDeliverEvent(rec)` to drop any that became superseded, handled, or dismissed while queued.
- Events are coalesced by `targetRunDir` so only the latest unresolved event per target is delivered, preventing redundant wake-ups for the same child.
- `markAttentionState(..., "seen")` is now applied only to the coalesced, actually-delivered events, not to the entire pending batch.

**Tests added to `packages/tango/src/attention.test.ts`:**
- `superseded event that was previously delivered is not deliverable` — validates re-filter drops stale queued events.
- `coalescing by target runDir keeps only latest unresolved event` — validates that three successive events for the same target leave only the latest deliverable.

## Risks / follow-up

- Attention file may grow unbounded; add compaction when record count exceeds threshold
- Pi extension does not yet auto-mark `handled` when `tango_result` is invoked via tool; relies on CLI `result`/`wait` to mark handled
- Dashboard attention APIs still derive from status metadata; replace with `listUnresolvedAttention` in a future node
